# two-player.app

Static web hub for quick family and two-player party games. The intended use is simple: open a game from a phone, cast or mirror it to a TV, and play with minimal setup.

Neon Feud is the current quality bar. Dirty Guesser is an early prototype and should not be treated as the target standard.

## Entrypoints

- `index.html` - Main game hub.
- `landing.css` - Styles for the hub.
- `neonfeud/index.html` - Neon Feud game entrypoint.
- `dirty_guesser_web/index.html` - Dirty Guesser game entrypoint.

There is no build step, package manager, or framework. This repo is plain HTML, CSS, JavaScript, JSON, and static assets. Use a local static server for development when testing `fetch()` calls, service workers, or audio behavior.

## Current Structure

- `neonfeud/`
  - `index.html` - Game UI and screens.
  - `styles.css` - Neon/TV-first visual system.
  - `script.js` - Game logic, state, scoring, timers, teams, store, custom packs, audio, and persistence.
  - `packs.json` - Built-in question packs.
  - `manifest.json` - PWA metadata.
  - `sw.js` - Service worker/cache setup.
  - `audio/` - Shared sound effects for game feedback.

- `dirty_guesser_web/`
  - `index.html` - Prototype game UI.
  - `css/styles.css` - Prototype styles.
  - `js/game.js` - Prototype game logic.
  - `data/scenarios.json` - Guessing prompts and answers.
  - `audio/` - Sound effects.

- Root files
  - `README.md` - Minimal project note.
  - `CNAME` - GitHub Pages custom domain config.
  - `llm.txt` - This guide for future coding agents.

## Product Direction

The site should feel like a small arcade of instant-play games for families, couples, and two-player couch competition. Prioritize:

- One-click launch from the hub.
- TV-readable layouts with large type, strong contrast, and clear game state.
- Phone-friendly controls and touch targets.
- Fast rounds, low explanation overhead, and visible scoring.
- Local static deployment with no required backend.

## Design Standard

Use Neon Feud as the design and interaction reference:

- Full-screen, high-contrast, landscape-friendly game surfaces.
- Big buttons, clear state changes, timers, scoreboards, rounds, and celebratory feedback.
- Strong visual identity per game without making the hub feel inconsistent.
- Audio/vibration feedback where useful, with a mute path.

Dirty Guesser needs a redesign before it represents the platform. Do not copy its small centered-card layout for new games unless the game specifically needs a compact phone-only experience.

## Game Module Convention

Keep each game self-contained in its own directory. A typical game should include:

- `index.html`
- game-specific CSS
- game-specific JavaScript
- JSON data if the game is content-driven
- local audio/assets used by that game

Avoid cross-game coupling unless a shared asset or helper is deliberately introduced. If shared code appears, keep it small and stable.

## Engineering Notes

- Use vanilla JavaScript unless a real need for a dependency emerges.
- Keep game state understandable and local to the game module.
- Prefer structured JSON for game content.
- Avoid placeholder code, placeholder copy, or half-wired controls.
- Preserve static hosting compatibility with GitHub Pages.
- Be careful with service workers and cache names when changing game assets.
- Test through a local static server, not only by opening files directly, because `fetch()` and service workers can behave differently.
